Workflow case study

agentic-ds in practice

Data projects with agents can get heavy fast: commands, files, claims, old decisions, half-remembered checks. agentic-ds treats zero error as the constraint and verification as the real bottleneck. The workflow keeps the next step visible, updates the checklist as the project moves, and protects the same three promises from beginning to end: transparency, reproducibility, and auditability.

Credit and sources

agentic-ds builds on Professor Scott Cunningham's GTD skill in MixtapeTools. That work showed a practical way to use agents with external memory, clear next actions, and a record outside the chat. This skill adapts that idea for data projects, where every important claim needs a path back to evidence.

This page is grounded in the hosted agentic-ds skill repo: SKILL.md, references/workflow.md, references/templates.md, and references/audit.md.

Why keep it simple

The point of the skill is not to make the user learn a private language of commands. The agent carries the machinery: read the state, surface what needs attention, propose one next step, and file the work as it happens. The project scope creates the checklist. The checklist guides the work. As the project changes, the checklist changes with it, so the next step is not hidden in memory or scattered across old messages.

The loop

  1. Read the state. Start from STATUS.md, CHECKLIST.md, the decision log, and the latest session log. The project should not depend on someone remembering the last session.
  2. Check for rot. Run freshness and drift checks before trusting old outputs. Stale files, missing traces, and unexplained table changes become visible work.
  3. Orient the user. Give a short since-your-last-visit brief: what the project is, what changed, what broke, and what needs judgment.
  4. Nudge. Surface gates, stale evidence, unexplained drift, and results that were mentioned but never filed.
  5. Propose one step. The next action should be the current unchecked gate, not a menu of possibilities. The user can redirect, but the checklist never silently skips a gate.
  6. File as you go. Decisions, findings, claims, and caveats are written when they happen. Conversation helps the work move; files preserve it.
  7. Close with memory. End with a session log, checklist update, regenerated status, and a Socratic self-check that the files still tell the truth.

The six habits

Decide first

Write the question, metric, population, comparison, baseline, and kill condition before results have a chance to flatter the method.

Script numbers

No important number gets to live only in chat. Evidence comes from numbered scripts and saved outputs.

File results

A finding is real when it is filed with provenance. Conversation is useful, but it is not memory.

Challenge claims

No claim is supported until it survives a test that could have killed it.

Open and close

Every session starts from the project state and ends by leaving the next session a better one.

Keep human gates

The agent can draft, compute, and check. The human still signs the spec, decisions, supported claims, and final story.

Why this matters here

Reproducible

A stranger can follow the trail from public data to scripts to tables to site JSON. The prediction scoreboard is regenerated, not edited by hand.

Transparent

The site says when the model failed, what replaced it, where the data can lag, and which outputs are experimental.

Auditable

The claims register, findings, decision log, source traces, session logs, and dashboard let someone inspect the work while it is still alive.

The final audit

The skill makes the last gate explicit: a final project audit should be adversarial, preferably in a fresh session, and should not quietly fix the work it is judging. The auditor traces claims to the register, checks the spec against the final artifact, recomputes headline numbers independently, and writes permanent audit artifacts under 07_final/audit/.

What about Claude and Codex?

More than one agent helped build this project. That is not something to hide. It is the reason the workflow matters. Claude, Codex, or any future agent can contribute code and copy, but the project cannot live in an agent's memory. It lives in the brief, spec, decisions, claims, findings, scripts, and dashboard.

The result

WC26 Pulse is not only a prediction site. It is an attempt to make the work inspectable: decide before analyzing, script every number, challenge every claim, keep failures visible, and let the human sign the gates.
