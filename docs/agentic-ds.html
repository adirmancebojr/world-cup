Workflow case study

agentic-ds in practice

Data projects with agents can get heavy fast: commands, files, claims, old decisions, half-remembered checks. agentic-ds is meant to make that work less overwhelming. The workflow keeps the next step visible, updates the checklist as the project moves, and protects the same three promises from beginning to end: transparency, reproducibility, and auditability.

Credit and sources

agentic-ds builds on Professor Scott Cunningham's GTD skill in MixtapeTools. That work showed a practical way to use agents with external memory, clear next actions, and a record outside the chat. This skill adapts that idea for data projects, where every important claim needs a path back to evidence.

This page is also grounded in the local agentic-ds skill files: SKILL.md, references/workflow.md, and references/templates.md.

Why keep it simple

The point of the skill is not to make the user learn a private language of commands. Anyone should be able to learn it by using it. The project scope creates the checklist. The checklist guides the work. As the project changes, the checklist changes with it, so the next step is not hidden in memory or scattered across old messages.

The loop

  1. Define the scope. Name the question, the data, the output, and the standard of trust before the work starts running ahead.
  2. Build the checklist. The checklist is not decoration. It is the map: what has to happen, what has happened, and what still needs judgment.
  3. Read the state. Start from STATUS.md, CHECKLIST.md, the decision log, and the latest session log. The project should not depend on someone remembering the last session.
  4. Move one step. The workflow should make the next action plain. Run the current step, record what changed, and resist the urge to turn everything into a new branch of work.
  5. Verify before claiming. A number is not trusted because an agent produced it. It is trusted when the script, input, check, and decision can be inspected.
  6. Update the checklist. Progress changes the project. The checklist has to change too, so the work stays guided instead of becoming a pile of finished and unfinished thoughts.
  7. Close with memory. End with a session log, checklist update, regenerated status, and a check that the files still tell the truth.

The six habits

Decide first

Write the question, metric, population, comparison, baseline, and kill condition before results have a chance to flatter the method.

Script numbers

No important number gets to live only in chat. Evidence comes from numbered scripts and saved outputs.

File results

A finding is real when it is filed with provenance. Conversation is useful, but it is not memory.

Challenge claims

No claim is supported until it survives a test that could have killed it.

Open and close

Every session starts from the project state and ends by leaving the next session a better one.

Keep human gates

The agent can draft, compute, and check. The human still signs the spec, decisions, supported claims, and final story.

Why this matters here

Reproducible

A stranger can follow the trail from public data to scripts to tables to site JSON. The prediction scoreboard is regenerated, not edited by hand.

Transparent

The site says when the model failed, what replaced it, where the data can lag, and which outputs are experimental.

Auditable

The claims register, findings, decision log, source traces, session logs, and dashboard let someone inspect the work while it is still alive.

What about Claude and Codex?

More than one agent helped build this project. That is not something to hide. It is the reason the workflow matters. Claude, Codex, or any future agent can contribute code and copy, but the project cannot live in an agent's memory. It lives in the brief, spec, decisions, claims, findings, scripts, and dashboard.

The result

WC26 Pulse is not only a prediction site. It is an attempt to make the work inspectable: decide before analyzing, script every number, challenge every claim, keep failures visible, and let the human sign the gates.
